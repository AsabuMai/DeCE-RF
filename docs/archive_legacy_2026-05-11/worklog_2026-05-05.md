# RF h-Edit Worklog, 2026-05-05

This records the latest pass on making the RF/ODE editing prototype more
general than the original panda-sunglasses setting. The main theme was moving
from a single generic edit field toward a more explicit decomposition:

```text
dx/dt = v_src + u_rec + u_struct + u_app
```

where `u_struct` handles structural/local generation and `u_app` handles
appearance/color changes. The current implementation is still experimental, but
the results clarify which parts are promising and which parts remain weak.

## 1. Support Rewriting: Anchor vs Edit Region

The earlier semantic support path used the grounded semantic mask directly as
`M_edit`. This caused failures such as cat/crown changing the cat face, because
`cat head` was both the localization anchor and the editable region.

This was changed to separate:

```text
M_anchor: source structure used for localization
M_support: actual editable support derived from M_anchor
```

Implemented in:

```text
scripts/make_semantic_mask.py
scripts/run_semantic_velocity_edit.sh
tests/test_mask_and_box_helpers.py
```

New support relations:

```text
inside
box
upper
lower
above
below
left
right
around
auto
```

The most important new helper is:

```text
support_from_anchor_mask(anchor_mask, relation, ...)
```

For insertion edits, the anchor and edit support can now differ. Example:

```text
cat head + above -> crown support above the head
panda eyes + box -> expanded glasses support around the eyes
```

## 2. Auto Support Planner

Added a conservative rule-based planner that maps target-prompt changes into an
anchor phrase and a spatial relation. This is not intended as the paper
contribution; it is a default support provider for experiments.

Examples:

```text
crown / hat / helmet       -> head + above
sunglasses / glasses       -> eyes + expanded box
scarf / necklace / collar  -> neck + around
shirt / jacket / dress     -> torso + inside
shoes / boots              -> feet + inside
color words                -> source subject + inside
```

The wrapper now defaults to:

```bash
SUPPORT_RELATION=auto
```

Important metadata written by `make_semantic_mask.py`:

```text
semantic_phrase_inferred
support_relation_requested
support_relation_inferred
support_plan_rule
support_plan_params
anchor_mask_area_ratio
mask_area_ratio
```

Default auto support was tested with:


Result: the crown is generated on the head while the cat remains side-facing.

The first auto rule used:

```text

```

This produced two black spots rather than complete sunglasses because the mask
covered only small eye islands. The glasses rule was changed to:

```text

```

Reference output:

```text
outputs/panda_sunglasses_auto_box_support_ode_20260504/result.png
```

Result: glasses are much more complete, including lenses and bridge, while the
mask is still derived automatically from the source image.

### More-Image Support Validation

Additional auto-support tests:

```text
outputs/dog_sunglasses_auto_support_20260504/result.png
outputs/corgi_crown_auto_support_20260504/result.png
outputs/rabbit_sunglasses_auto_support_20260504/result.png
outputs/auto_support_more_images_20260504.png
```

Readout:

- Dog + sunglasses worked: the semantic eye region was accurate enough.
- Corgi + crown worked: head/above support produced a plausible crown.
- Rabbit + sunglasses failed: `rabbit eyes` was grounded as a very large
  head/body region, and the later mask guard could not recover a useful local
  support.
- Tiger + sunglasses exposed the same issue at the mask stage: animal eye
  grounding can become too large and should trigger a fallback.

Current conclusion: the support planner is useful, but it needs quality gates.
For glasses-like edits, if the `eyes` anchor is too large, the planner should
fallback to a head/upper or RF velocity-refined support instead of trusting the
semantic detector.

## 4. Why Color Editing Needs A Separate Field

Color/appearance editing behaved differently from structural insertion.

Using the same `rf_text_diff` edit field for:

```text
yellow bus -> blue bus
yellow bulldog statue -> blue bulldog statue
```

showed that color changes can easily trigger structural rewriting. The bus
turned blue, but the front geometry was redrawn. The bulldog result was not a
stable color transfer and partially regenerated the object.

This motivated splitting the edit term conceptually:

```text
u_struct: structural / geometric / insertion edits
u_app: appearance / color / material edits
```

The method should not claim that `u_struct + u_app` covers every possible edit,
but it is a more general decomposition than task-specific fields such as
`u_crown`, `u_glasses`, or `u_color`.

## 6. Color-Specific Support

To avoid recoloring glass, tires, headlights, and text, color editing now uses:

```text
M_color = M_object ∩ similarity(source_image, source_color)
```

For `yellow bus -> blue bus`, the source color is inferred as yellow and the
target as blue. The color branch generates:

```text
masks/color_edit_mask.png
```

Reference outputs:

```text
outputs/bus_blue_color_masked_k012_20260505_retry/result.png
outputs/bus_blue_color_masked_k035_20260505/result.png
outputs/bus_blue_color_masked_k035_20260505/masks/color_edit_mask.png
```

Result: glass is no longer globally dyed blue. The editable region is much more
specific than the whole semantic object mask. However, front-face details can
still be affected because they are genuinely part of the yellow body region.

## 7. Chroma/Luma Color Loss

The RGB mean loss was replaced with a differentiable YUV-style loss:

```text
E_color =
  || UV(current) - UV(target_color) ||^2
  + lambda_luma || Y(current) - Y(source) ||^2
```

This is intended to push chroma while preserving source luminance and texture
better than a direct RGB mean target.

Reference output:

```text
outputs/bus_blue_color_yuv_k035_20260505/result.png
outputs/bus_color_mask_yuv_compare_20260505.png
```

Readout:

- The color direction is correct.
- The glass is better protected than in the full-mask RGB version.
- Some front-face details are still blurred or contaminated.

Current conclusion: `M_color` and chroma/luma control are the right direction,
but the next version needs an explicit edge/high-frequency preservation term
inside `M_color`, for example preserving source luminance gradients.

## 9. Current Method Claim After This Pass

The project should not be framed as a glasses/crown editing method. The more
defensible claim is:

```text
RF/ODE image editing can be decomposed into a source-conditioned base velocity,
a reconstruction/preservation correction, and modular edit controllers.
```

The current controller set is:

```text
u_struct: source-target RF/text edit field for structural or insertion edits
u_app / u_color: color/appearance field with color-specific support
```

The support provider is plug-in:

```text
attention_velocity
semantic
semantic_velocity
semantic + relation support
external masks
```

The semantic detector is not the novelty. It is a support proposal mechanism
that can be replaced by attention, RF velocity, external segmentation, or other
providers.

## 10. Known Weaknesses

- The auto support planner is still a conservative rule system; it is useful
  for experiments, but not a full semantic planner.
- Animal eye grounding is unreliable for some images and needs automatic
  fallback.
- `u_color` still lacks explicit edge/texture preservation.
- Color-specific support protects glass better, but body-region details can
  still be affected.
- `u_struct + u_app` is not enough for all editing types. Relation, pose,
  count, deletion/inpainting, and text editing likely need additional optional
  controllers.

## 11. Next Steps

Most immediate:

1. Add an edge/high-frequency preservation term to `u_color`, likely on source
   luminance gradients inside `M_color`.
2. Add quality gates for semantic support, especially for `eyes` anchors:
   fallback from `eyes + box` to `head + upper` or velocity-refined support
   when the grounded region is too large.
3. Run a small grid of color edits with the same `u_color` branch:
   `yellow -> blue`, `red -> blue`, `white -> black`, and material/texture
   changes if available.
4. Write a short method note describing the controller decomposition:

```text
dx/dt = v_src + u_rec + u_struct + u_app + optional controllers
```

and explicitly state that structural and appearance controllers are general
building blocks, not task-specific tricks.
