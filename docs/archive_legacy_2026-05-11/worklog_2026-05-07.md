## 6. Bus Color Detail Protection and Chroma Scaling

Implemented two additional controls for the bus color guidance path:

```text
--edit-color-detail-protect-scale
--edit-color-detail-protect-threshold
--edit-color-detail-protect-softness
--edit-color-target-chroma-scale
--edit-color-smooth-kernel
```

The detail-protect branch computes a luma edge map from the decoded source image
and downweights high-frequency regions inside `color_edit_mask`. This is meant
to reduce direct color pressure on tires, lamps, grille/window edges, and other
structure-heavy regions.

The target-chroma scale keeps the existing YUV color objective but scales the
target U/V channels before computing chroma loss:

```text
target_yuv[:, 1:] *= edit_color_target_chroma_scale
```

The color smooth-kernel control exposes the latent-space smoothing used on the
color guidance gradient. The old/default behavior is `5`; setting it to `1`
turns off this extra spatial smoothing for the color term only.

Runs:

```text
outputs/bus_blue_color_detailprotect080_k035_20260507/result.png
outputs/bus_blue_color_detailprotect080_k035_20260507/masks/color_edit_mask.png
outputs/bus_blue_color_chroma065_detailprotect080_k035_20260507/result.png
outputs/bus_color_detail_chroma_compare_20260507.png
outputs/bus_blue_color_chroma065_detail080_smooth1_k035_20260507/result.png
outputs/bus_color_detail_chroma_smooth_compare_20260507.png
```

Readout:

- `detail_protect=0.8` makes the mask more conservative around strong source
  edges, but does not materially fix the dirty blue/halo artifact by itself.
- `target_chroma_scale=0.65` makes the blue slightly less fluorescent while
  preserving the overall edit, but the front face still has gray/dirty regions.
- `edit_color_smooth_kernel=1` is the clearest improvement in this batch. It
  reduces color spill around the grille/lights/front-edge regions and preserves
  local structure better than the default `5`.
- The remaining issue is mask semantics: the current color mask still treats
  some non-paint parts of the front bus face as recolorable. The next useful
  step is a paint-body mask or exclusion mask for tire/lights/window/grille,
  then tune color strength on top of `smooth_kernel=1`.

## 12. Reference Appearance Guidance Inside ODE

Problem:

The masked color energy alone cannot reliably infer where vehicle paint color
should land on complex fronts. It tends to make the bus blue but also introduces
blur, dirty texture, or over-smooth areas because the guidance has only a
global color target and a mask.

Implemented an ODE-native reference appearance term instead of changing the
bottom-level RF/ODE structure:

```text
E_ref(x) =
  masked chroma distance to a reference image
  + luma preservation inside the same mask
  + gradient/detail preservation inside the same mask
```

The new term is injected as another edit velocity component, alongside the
existing reconstruction, trajectory-preservation, text, HEdit, and color
guidance terms. The ODE sampler and velocity-composition logic remain the same.

Added interface:

```text
EDIT_REF_GUIDANCE_SCALE
EDIT_REF_IMAGE
EDIT_REF_MASK
EDIT_REF_LUMA_PRESERVE_SCALE
EDIT_REF_GRADIENT_PRESERVE_SCALE
EDIT_REF_SMOOTH_KERNEL
```

Changed files:

```text
sd3_hrec.py
run_edit_sd3.py
scripts/run_ode_decoupled_edit.sh
```

Validation runs:

```text

outputs/bus_blue_edit_ref_guidance_blue_ref_s070_20260507/result.png
outputs/bus_edit_ref_guidance_compare_20260507.png
```

## 13. Stabilized Reference Velocity Injection

Added controls for how the reference appearance velocity enters the existing
ODE update:

```text
EDIT_REF_SCHEDULE_START
EDIT_REF_SCHEDULE_STOP
EDIT_REF_SCHEDULE_POWER
EDIT_REF_MAX_STRUCT_RMS_RATIO
EDIT_REF_PROJECT_STRUCT_CONFLICT
```

Math:

```text
v_ref <- ramp(t) * v_ref
v_ref <- remove component of v_ref that opposes v_base + v_rec
v_ref <- clamp RMS(v_ref) <= ratio * RMS(v_base + v_rec)
```

This keeps the RF/ODE decomposition unchanged. It only stabilizes the new
reference energy's velocity injection.

- `s1.15_stable3` is the current best ODE-internal balance: more blue than

  local cyan/green/purple artifacts.

  matching with structure/luma preservation.

Reference generator update:

```text
scripts/make_surface_recolor_reference.py --mode hsv-target
```

`hsv-target` directly injects the target color H/S and preserves source V,
then blends at image level. This avoids the old `hsv-hue` behavior where
yellow-to-blue hue interpolation could pass through green at soft mask edges.
It is now the default reference mode; `hsv-hue` remains available for ablation.

Current recommendation:

- Use `EDIT_REF_CHROMA_MODE=yuv` by default for complex or reflective surfaces.
- Use `EDIT_REF_CHROMA_MODE=yuv_direction` for simple large color regions when
  absolute YUV matching produces a dark/weak result.

## 24. Main Matrix Mask Provider Switched Back to SAM/Semantic

The first `run_main_table.sh` version used the generic no-oracle provider:

```text

```

This was useful as a baseline, but it did not reflect the later mask strategy
from the previous experiments. The main runner now defaults to:

For each run it generates:

using:

```text

```

and then passes:

```text
--object-mask-provider semantic_velocity
--support-mask masks/semantic_base_generated.png
```

