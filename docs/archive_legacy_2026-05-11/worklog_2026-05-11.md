# Worklog 2026-05-11

## 1. Generic Support Direction

Continued the RF h-Edit direction from task-specific support rules toward:

```text
generic support proposal + clean-estimate-space adaptive controller
```

The motivation is that the previous good results relied heavily on manual or
task-specific support rules, such as glasses boxes, decal boxes, headwear
relations, and removal-specific masks. This makes the method harder to present
as a general RF editing method.

The current framing is:

- Generic support proposer estimates where editing should happen.
- Adaptive controller decides whether edit progress is enough and whether
  non-edit regions are drifting.
- RF velocity correction executes local edit and preserve control during the
  ODE trajectory.

## 2. Generic Support Implementation

Added:

```text
generic_support.py
```

Main functions:

- normalize spatial maps.
- compute clean-estimate disagreement.
- compute velocity disagreement.
- build support score.
- postprocess support score with percentile thresholding, component filtering,
  area guard, dilation, and blur.
- generate edit/core/preserve support statistics.

Supported score modes:

```text
attention_only
clean_disagreement_only
velocity_disagreement_only
attention_x_clean
attention_x_velocity
```

Default support score:

```text
S = Norm(A_changed) * Norm(D_clean)
```

where `A_changed` is changed-token attention and `D_clean` is disagreement in
clean-estimate space.

## 5. Result Readout

The result is mixed.

Strong point:

- Structure preservation is good.
- The clean-estimate adaptive controller keeps non-edit regions stable.
- `dog_sunglasses` is the strongest generic-support case.

Weak points:

- `cat_crown` often fails to create a clear crown.
- `mug_heart` does not reliably produce a clean heart without decal/reference
  evidence.
- `backpack_remove_toy_charm` remains hard for generic removal support.

Current interpretation:

```text
The adaptive controller is useful when support is reasonable, but the generic
support proposer is not yet strong enough to fully replace high-quality manual
or external support.
```

This means the next method improvement should focus on support quality rather
than adding more controller terms.

## 18. Controller Failure Diagnosis

Problem being investigated: why adaptive v0/v1 do not reliably beat the fixed
support-v3 controller, especially on `mug_heart`.

Main finding:

```text
The clean-effect projection described in the controller summary is not actually
applied to v_edit_total.
```

Evidence:

```text
docs/rf_h_edit_mainline_controller_summary.md describes:
  clean edit effect = -t * u_edit
  project that clean effect away from preserve error
  map the projected clean effect back to u_edit

sd3_hrec.py currently computes:
  clean_conflict_dot
  destructive_effect
  clean_effect_projected
  adaptive_preserve_drift_after_projection_estimate

But the projected clean effect is only used for the diagnostic estimate.
The actual velocity update then calls remove_masked_opposing_guidance_component
with anchor = v_base + v_rec.
```

This means the code diagnoses one conflict but removes a different one:

```text
diagnosed conflict:
  < preserve_error, -t * v_edit_total > > 0

actual removed component:
  component of v_edit_total opposing v_base + v_rec
```

Per-step logs confirm the mismatch:

```text
dog_sunglasses v1 early steps:
  clean conflict score reaches 46-53,
  but adaptive_projection_norm remains 0.00.

mug_heart v1 early steps:
  clean conflict score is often positive,
  but projection_norm is usually 0.00 or tiny.

Reason:
  remove_masked_opposing_guidance_component only removes a component when the
  velocity-space dot with v_base + v_rec is negative. Positive clean-space drift
  damage does not necessarily produce that negative velocity-space dot.
```

Secondary finding:

```text
adaptive_preserve_weight only scales reconstruction guidance.
It does not currently scale trajectory_preserve_scale, source anchor correction,
or a direct preserve-region clean correction, even though the summary lists
those as intended preserve-lock targets.
```

This explains the current behavior:

```text
1. preserve drift is already high early in the trajectory;
2. adaptive_preserve_weight quickly rises;
3. rec guidance norm increases;
4. but the harmful clean-space component of edit guidance is not actually
   projected out;
5. final outside-mask L1 therefore stays similar to fixed, or gets worse when
   adaptive edit boost increases semantic pressure.
```

Concrete next implementation step:

```text
Replace the current adaptive projection application with a real clean-space
projection:

  clean_edit_effect = -t * v_edit_total
  clean_effect_projected = clean_edit_effect - destructive_effect
  v_edit_total_projected = -clean_effect_projected / t

Apply this projected velocity inside the preserve gate, keep the original edit
velocity outside the preserve gate, then optionally keep the existing
velocity-anchor projection only as an additional diagnostic or ablation.
```

Expected experiment after the fix:

```text
Run the same fixed-support matrix:
  TASKS="P1 P2 P3" METHODS="M17 M18 M19" SEEDS=10 ...

Primary pass condition:
  mug_heart outside_mask_l1 should move down from 0.0206-0.0225 toward <0.015
  without losing positive CLIPDelta.
```
