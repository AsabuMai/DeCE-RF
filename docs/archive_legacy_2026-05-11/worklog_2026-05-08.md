## 5. Technical Issues Found

### Mask-provider confusion

The project has three distinct mask paths:

- `attention_velocity`: generic no-oracle baseline.
- `semantic_velocity`: GroundingDINO/SAM support + RF velocity refinement.
- `final_edit_mask`: trusted late override.

The main runner now uses `semantic_velocity` by default, but old seed-10 runs
mostly used `attention_velocity`. Do not mix these as the same method in paper
tables.

### Headwear support too coarse

Current headwear logic:

```text
SAM head anchor -> support_relation=above -> horizontal band above head bbox
```

For cats, the head bbox includes ears, so the crown can overlap or attach to
the ears.

Needed next:

- Add a `top_center` support relation for crown/hat/headwear.
- It should use the SAM head anchor but create a narrower centered band above
  the head, instead of a full-width band spanning ear tips.

### Color edit route still incomplete

Prompt/target velocity is not enough for clean color edits. The color tasks
should use:

```text
surface/object mask
make_surface_recolor_reference.py
EDIT_REF_IMAGE / EDIT_REF_MASK / EDIT_REF_STRUCTURE_IMAGE
EDIT_REF_CHROMA_MODE=yuv
```

before expanding to more seeds.

## 7. Submission Gap Assessment

Current shortest path to a publishable paper is not to add more exploratory
cases. It is to turn the existing prototype into a fixed, fair, reproducible
experiment package.

Largest gaps:

1. Main experiment matrix is not complete.
   `experiments/main_matrix.md` exists, but the matrix is not systematically
   filled. A paper needs fixed tasks, fixed methods, fixed seeds, and completed
   results, not only exploratory case piles.

2. Formal metrics are missing.
   Current judgment is mostly visual comparison plus internal `stats.json`.
   At minimum the paper needs:
   - edit success: CLIP, VLM, or task-specific score.
   - preservation: LPIPS, DINO, SSIM, or outside-mask L1.
   - mask outside drift.
   - runtime and memory.
   - failure rate.

3. Baseline parity is not enough.
   RF-Solver, FireFlow, FlowEdit, ReFlex, and h-Edit cannot only be mentioned
   or run under weaker settings. They need comparable input, prompt,
   resolution, seed, and mask conditions. Otherwise reviewers can argue the
   comparison is unfair.

4. Paper materials are not really started.
   The project has `paper/`, but does not yet have a usable `main.tex`,
   `draft.md`, or `references.bib`. Therefore the academic-pipeline cannot
   truly enter writing/revision stage yet; it can only give research gap
   assessment.

5. Reproducibility package is incomplete.
   Existing scripts are useful, but a submission needs one-command or clearly
   staged scripts:
   - `scripts/run_main_table.sh`
   - `scripts/run_ablation_table.sh`
   - `scripts/make_paper_figures.sh`
   - `environment.yml` or `requirements.txt`

6. Method boundary needs tightening.
   Recent color-editing notes exposed an important framing issue: some modules
   are engineering utilities under the general theory, not the main theoretical
   contribution. The paper must clearly separate the main RF dynamics from
   mask/reference utilities, otherwise the method can look unfocused.
